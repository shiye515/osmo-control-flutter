## Context

根据 DJI R SDK 协议文档：
- **休眠命令**：CmdSet=0x00, CmdID=0x1A, powerMode=3
- **唤醒流程**：相机休眠后无法接收 BLE 数据，需要通过 BLE 广播唤醒数据 2 秒

唤醒广播数据格式：
```
[10, 0xFF, 'W', 'K', 'P', MAC地址（反转）]
```

前提条件：
- 遥控器近期连接过该相机
- 相机进入休眠状态未超过 30 分钟

## Goals / Non-Goals

**Goals:**
- 实现正确的 BLE 广播唤醒流程
- 添加调试台的休眠/唤醒按钮
- 支持平台判断（iOS 可能不支持）

**Non-Goals:**
- 不修改休眠命令实现（已正确）
- 不实现自动唤醒逻辑

## Decisions

1. **唤醒方式选择**
   - Android：使用 `flutter_blue_plus` 的广播 API
   - iOS：由于系统限制不支持自定义广播，提示用户手动唤醒相机
   - 理由：iOS 不支持 BLE Peripheral 模式的自定义广播数据

2. **广播时机**
   - 用户点击唤醒按钮时开始广播
   - 广播持续 2 秒后自动停止
   - 理由：协议规定广播 2 秒足够唤醒相机

3. **UI 设计**
   - 在调试台添加休眠/唤醒按钮
   - 休眠按钮发送命令后，相机断开连接
   - 唤醒按钮触发广播，等待相机重连
   - 理由：调试台适合这类低频操作

## Risks / Trade-offs

- **Risk**: iOS 用户无法使用唤醒功能
  → **Mitigation**: 显示提示信息，建议手动按相机电源键唤醒

- **Risk**: 广播可能需要特定权限
  → **Mitigation**: 在 AndroidManifest 中添加必要权限
