## Context

当前 Flutter 移动应用使用自定义的 DUML 协议实现与 DJI Osmo 设备的蓝牙通信，但该实现与官方 DJI R SDK 协议完全不兼容：

| 对比项 | 当前实现 | DJI R SDK 协议 |
|--------|----------|----------------|
| 帧头 SOF | 0x55 | 0xAA |
| Write UUID | FFF6 | FFF5 |
| Notify UUID | FFF7 | FFF4 |
| 拍录控制 | (0x01, 0x4B) | (0x1D, 0x03) |
| CRC 校验 | CRC-8 + CRC-16 | CRC-16 + CRC-32 |
| 连接流程 | 无鉴权 | 需要鉴权握手 |

**官方文档来源**: https://github.com/dji-sdk/Osmo-GPS-Controller-Demo/blob/main/docs/

## Goals / Non-Goals

**Goals:**
- 实现完整的 DJI R SDK 协议帧封装与解析
- 实现连接请求鉴权流程（首次配对和已配对）
- 修正所有控制命令使其与官方协议一致
- 实现相机状态订阅机制
- 将协议文档写入项目便于维护

**Non-Goals:**
- 不支持加密模式（官方协议支持 AES256，但当前不需要）
- 不实现多相机连接（暂只支持单机）
- 不修改 Web 端实现（Web 端不涉及蓝牙）

## Decisions

### D1: 协议架构分层

采用三层架构，与官方 Demo 结构一致：

```
BLE Layer (ble_service.dart)
  ↓ 发送/接收原始字节
Protocol Layer (dji_protocol.dart)
  ↓ 封装/解析协议帧
Data Layer (dji_data_processor.dart)
  ↓ 构造/解析 DATA 段
Logic Layer (providers/)
```

**理由**: 分层架构便于扩展新命令集，各层职责清晰，与官方 Demo 保持一致便于对照调试。

### D2: 序列号管理

全局递增序列号，模 65536 循环。每次发送命令帧生成新序列号，接收应答帧需匹配序列号。

**理由**: 官方协议要求应答帧使用相同序列号，便于追踪请求-应答配对。

### D3: CRC 校验实现

- CRC-16: 校验 SOF 到 SEQ 段（前 10 字节）
- CRC-32: 校验 SOF 到 DATA 段（整帧减最后 4 字节）

使用官方 Demo 中的 `custom_crc16.c` 和 `custom_crc32.c` 算法。

**理由**: 必须与相机端 CRC 实现一致，否则帧会被拒绝。

### D4: 连接请求流程

```
遥控器 → 相机: 连接请求 (verify_mode=0 或 1, 包含校验码)
相机 → 遥控器: 连接应答 (立即返回)
相机 → 遖控器: 连接请求 (verify_mode=2, 包含校验结果)
遥控器 → 相机: 连接应答 (包含相机编号)
```

首次配对时 `verify_mode=1`，相机弹窗显示校验码；已配对时 `verify_mode=0`，相机自动判断。

**理由**: 官方协议强制要求此流程，未完成鉴权前其他命令无效。

### D5: 相机状态订阅

连接成功后发送订阅命令 (push_mode=3, push_freq=20)，启用周期+状态变化推送，频率固定 2Hz。

**理由**: 只有订阅后相机才会推送状态，否则无法获取相机模式、录像时间等信息。

### D6: 休眠唤醒机制

相机休眠后通过 BLE 广播唤醒，广播数据格式: `[10, 0xFF, 'W','K','P', MAC地址(反转)]`，持续 2 秒。

**理由**: 休眠状态下 BLE 连接断开，只能通过广播唤醒。30 分钟内已配对设备可唤醒。

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|----------|
| CRC 实现差异导致帧被拒绝 | 使用官方 Demo 的 CRC 算法，单元测试覆盖 |
| 相机固件版本差异 | 添加版本查询命令，记录固件版本 |
| 连接请求超时 | 设置 10 秒超时，提示用户重新操作 |
| 状态推送解析错误 | 完整实现 1D02 和 1D06 命令解析 |
| Flutter BLE 平台差异 | 使用 flutter_blue_plus 跨平台库 |

## Migration Plan

1. 创建新的协议文件，保持旧文件暂不删除
2. 替换 BLE UUID 配置
3. 更新 Provider 中的连接和控制逻辑
4. 测试连接、控制、状态订阅
5. 确认功能正常后删除旧协议文件
6. 更新文档
